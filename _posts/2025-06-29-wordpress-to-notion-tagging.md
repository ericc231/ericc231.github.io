搬站，是每位寫作者遲早都會面對的課題。而我從 WordPress 搬進 Notion，不只是單純的搬家，而是一場「知識結構重整」的過程。

### 🚚 為什麼搬到 Notion？

長年在 WordPress 上寫文章，儘管架構完整、插件齊備，但幾年下來我還是遇到幾個痛點：

- 想要跨文關聯、標籤分類但 WordPress 做起來不夠直覺
- 每篇文章的 metadata（分類、標籤）使用自由度高，但久了就變「混亂」
- 想要同一篇文章，同時對應到閱讀路徑、主題分類與情緒標記，但缺乏維度管理工具

Notion 的資料庫功能，意外成為理想的替代方案：區塊式結構、屬性定義彈性、還能透過視圖做不同切角的內容整理。

但很快我發現：**從 WordPress 遷移到 Notion 的過程，最關鍵的並不是「匯入」，而是「貼標」。**

### 🔄 我原本打算用 Notion API，但放棄了

當時第一個念頭當然是走 Notion API，把所有文章資料自動寫入 database。不過很快遇到幾個阻力：

- API 欄位 mapping 太細，文章內容如果很長，分段與 rich text 控管很煩
- 匯入的 metadata 要額外轉格式
- 還要自己處理 error handling 與 token refresh

結果我選擇走最簡單的方案——**用 Notion 的 CSV 匯入**。乾淨、好用、不容易壞。

### 📁 匯入文章內容與欄位設計

我整理出一份 `articles.csv`，欄位大概如下：

- `title`：文章標題
- `slug`：原 WordPress 的網址別名
- `date`：發文日期（可選）
- `tags`：貼標結果（逗號分隔）
- `content`：純文字文章內容

其中最花時間的是 tags。我希望它不是單純的「分類」，而是有語意的資訊單位。這就牽涉到一整個貼標流程（我會在另一篇詳談），但簡單說，我用了：

- KeyBERT + 中文模型產出候選關鍵字
- 自建清洗機制，排除語助詞與「出發前」「這次旅行」等語意過渡詞
- 部分使用 LLM 做 semantic QA，判斷標籤是否具資訊性

這樣我最後產出的 `tags` 欄位內容更像這樣：

`美國國家公園, 公路旅行, 大峽谷, 攝影紀錄`

這些資料就能讓 Notion 的多選欄（multi-select）完美對應。

### 🛠️ 實作：標籤回寫與 CSV 組裝

我寫了一支 `write_back_tags.py` 的工具，把標籤結果寫進我準備要匯入的 `articles.csv`。每一筆文章就長得像這樣：

| slug | title | tags |
|---|---|---|
| grand-canyon | 大峽谷紀錄與觀察 | 美國國家公園, 壯麗風景, 旅行, 攝影 |

這樣一來，我只需要：

1. 開 Notion，新建一個 database
2. 匯入這個 CSV
3. 把 `tags` 欄設成「多選欄位」
4. Done！🔥

### ✨ 結語：貼標是搬站的靈魂

這次從 WordPress 搬家到 Notion，我不只是「搬了文章」，而是趁機為自己的內容資料重新「加上語意索引」。

你會驚訝地發現——當你把標籤處理好，所有內容都變得有組織、可查詢、可視覺化，甚至可以開始做自己的 Notion Home。

我會在另一篇文章分享貼標流程的完整進化路線圖，從關鍵字提取、stopwords 清洗、語助詞剃除，到最後導入 LLM 審查，一步步建立起「語意貼標治理中樞」。

想把自己的筆記系統變成個人知識圖譜的話，這套做法也許會給你不少啟發。
