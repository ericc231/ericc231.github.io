在我開始將自己的文章搬進 Notion 前，我從沒想過「貼標」這件事，會這麼麻煩。

回顧 WordPress 上面文章的標籤列表，我開始理解問題有多大。重複的標籤、模糊的分類、語助詞型的空洞詞彙──它們就像資訊架構中被遺忘的雜訊，一邊阻礙我找資料，一邊模糊我過去寫作的主題結構。此時我問了自己一個問題，「這麼多文章裡，我談過什麼真正重要的主題？」

為了回答這個問題，我決定自己打造一套語意貼標的治理系統，不受情緒左右，能忠實的依照文章的內容摘要出最精要的描述。一路上從傳統 TF-IDF，到語意相似度，再到大型語言模型（LLM）的 QA 審查，這個系統不斷進化──只為了把「分類」這件事做好。

## ✨ 貼標混亂的痛感，是這整個系統的起點

WordPress 上的標籤，大多是隨性決定的一串詞。但這些詞逐漸演變成：

- 出現十幾次的「這次旅行」、「整體心得」這類無資訊詞
- 意義重複、差一兩字但無法聚合的標籤（例如：「閱讀反思」與「閱讀心得」）

在這次將文章轉移到 Notion 的過程中，我決定打造一個理性不受心情、天氣等雜訊影響的貼標系統。

接著我將分享我這次的「貼標進化路線圖」，怎麼一步步的從雜亂的擷取標籤走向明確分析語意結構並透過 LLM 檢視結果。

## 🛠️ 貼標系統的進化過程：從詞頻到語意邏輯

這整段打造過程，就像給標籤治理開了一場「機能升級」手術。一開始我想的只是簡單的關鍵字擷取，結果一路長出了多模型融合、語助詞剃除、後綴策略與語意審核模組⋯⋯最後變成一個比想像更完整的「貼標中樞系統」。

以下是我的貼標技術路線圖，每一步都是為了解決前一步的不足：

| 世代階段 | 核心方法 | 為什麼不夠用 |
|---|---|---|
| 初代 | KeyBERT（關鍵字擷取） | 過度仰賴詞頻與表面相關，標籤資訊密度不足 |
| 二代 | 加入 CKIP-BERT 模型（繁中語意） | 雙模型輸出彼此無法整合，結果分散 |
| 三代 | 多模型輸出融合（union 策略） | 有效標籤進入增加，但同時噪音也大量進入 |
| 四代 | 加入 POS（詞性）、停用詞清洗 | 成效提升，但對語助詞與語意弱詞無法判別 |
| 五代 | 語助詞後綴剃除 + similarity 過濾 | 處理邏輯語意好轉，但仍需要語境與主題判別 |
| 六代 | LLM 審查「是否抽象／主題無關／過渡詞」 | 回應語意更全面，但也引出 LLM 統一性挑戰 |

從這段進化史你可以看見，其實我們的分類困境從來不是模型沒找到正確詞，而是——**它找到了太多模糊的詞**。而語助詞、過渡語、抽象詞⋯⋯這些都是讓分類失準的隱性殺手。

## 🤖 為什麼 LLM 不適合主導貼標，而適合擔任「語意審查員」？

在很多人看來，把貼標交給大型語言模型（LLM）應該是最聰明的選擇。畢竟它能理解語句、具備上下文能力、甚至能根據 prompt 調整輸出風格。

但實際下場試了之後，我的結論是：

> LLM 的語言能力沒有問題，但它缺乏「貼標邏輯的一致性」。

舉個例子：「出發前」這種詞語，從語法上沒問題、語氣也常見，但從分類角度來看，幾乎沒有資訊密度。它更像是一個語意過渡的「段落開場詞」，不應被納入標籤。

但 LLM 常常不會過濾它，甚至還會保留一些更抽象的字眼，像是：

- 經驗
- 這次旅程
- 某些東西
- 狀態
- 部分

當我反覆地嘗試不同 prompt 提示，加入規則說明、例子範本、甚至極端排除條件，LLM 還是會間歇性地留下不該留下的東西。

我開始意識到，這其實是 LLM 的優點之一：**它會根據語境給出彈性的解釋。** 但這對「貼標系統」來說，反而是最大的風險。

### 🧭 語意治理需要的是「邏輯可預期」，而非「語感靈活」

標籤不是散文，它更像一組分類邏輯的應用實例。而分類，講求的是一致性、可分群性，以及最重要的：

- 能不能幫我有效篩選出主題相近的文章？

當我使用規則＋語意分析（像是 TF-IDF、語助詞清除、語意相似度）來處理時，系統行為是可以預測的。我知道為什麼某個詞會被剃除，也知道它被留下的邏輯。

因此，我最後把 LLM 的角色重新定位成「語意 QA 審查員」，讓它來幫我判斷：

- 某個標籤是否太抽象？
- 是否只是過渡語？
- 是否與主題毫無關聯？

我甚至設計出一組結構化格式的回應要求，像這樣：

```json
{
  "should_filter": true,
  "reasons": ["抽象", "語助詞"],
  "explain": "該詞為常見語意過渡詞，缺乏資訊密度"
}
```

這樣一來，我仍然保有語意治理的邏輯一致性，也得到了 LLM 在語感、上下文與微妙判斷上的補強能力。

## 🧩 我現在的做法：語意貼標 + LLM 審核的雙軌機制

經過一輪又一輪的實驗，我發現貼標若要有一致性，又能反映文章的實際主題，不可能只靠模型單打獨鬥。

我最後採用的，是一種結合多模型語意擷取 + 自訂清洗規則 + 語意 QA 審查的「貼標治理中樞」，整個邏輯大致如下：

### 🔹 處理流程概覽

1. **多模型貼標融合**：使用 KeyBERT、CKIP-BERT（繁中）、zh-embed 產生候選語意標籤，並透過 union 或權重策略進行整合
2. **語意清洗與去雜訊**：自建停用詞清單、剃除語助詞後綴（如：「部分」「行程」）、排除 POS 結構為助詞 / 介詞 / 虛詞的標籤
3. **語意相似度過濾**：將標籤與標題計算語意距離，排除那些與主題無關的詞
4. **LLM QA 審查層**：最後交給 LLM 分析每個標籤是否抽象、語助詞、主題無關，並以結構化方式回應，統一審查規則

### 🔍 範例審查記錄

| tag | 審查結果 | 剃除原因 | LLM 說明（摘要） |
|---|---|---|---|
| 出發前 | 剃除 | 語助詞、抽象 | 常見段落轉場語，無具體語意，分類貢獻低 |
| 大峽谷 | 保留 | — | 主題明確地點詞，能有效建立內容分類依據 |
| 這次旅程 | 剃除 | 抽象、無主題關聯 | 缺乏語意資訊，僅表述經驗本身 |
| 攝影紀錄 | 保留 | — | 精準描述主題方向，具可聚合性與範疇性 |

你可以看到這樣的方式，一方面維持了治理規則的一致性，另一方面也讓貼標結果更有語意層次。

最重要的是，當回過頭檢視標籤資料庫時，我已經可以從「分類原則」去選擇標籤，而不是從「寫作當下的心情」。這對於內容再組織與關聯拓展，有極大的幫助。

## 🎯 結語：貼標的終點，其實是語意治理的起點

從一開始面對雜亂無章的 WordPress 標籤，到最後建構出一個結合多模型、語意清洗、後綴策略與 LLM 審查的完整貼標流程，我體會最深的一件事是：

> 貼標不是一個模型的問題，而是一個策略的設計問題。

當我們期待某個 AI 模型「自動幫我找出這篇文章的主題」時，其實更該問的是：

- 這篇文章的標籤，是要幫我達成什麼目的？
- 是幫我分類？幫我搜尋？還是幫讀者導航？

沒有這個目的，貼標的模型選得再好，最後也只是資料庫裡的一堆詞。而當我們開始思考「哪種類型的詞該保留、哪一類詞不能保留」時，我們就已經踏進了「語意治理」的設計領域。

我現在的貼標流程，並不完美，但它讓我：

- 明確知道每個標籤被留下或被移除的理由
- 隨時可以更新規則與詞庫，讓系統不斷進化
- 遇到模稜兩可的例子時，有結構化的方式讓人與機器共同判斷

對我來說，這就足夠了。因為真正的分類系統，不是只有機器能用的黑箱，而是人與機器能共同理解的語意地圖。

你也正面對內容貼標的混亂嗎？你可以從一組清洗後的停用詞表開始，也可以嘗試給你的標籤下一個真正有意圖的邏輯。只要你開始這樣做，就已經不是在「貼標」了，而是在設計自己的知識分類法了。
