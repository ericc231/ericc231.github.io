這篇文章要講的，不是 NLP 工具怎麼用、哪個模型效果比較好，而是我一路走來為了解決分類混亂、標籤語意模糊問題，逐步打造出的一整套治理系統。

在前兩篇文章中，我談了：

- 如何將 WordPress 搬進 Notion，並建立一套穩定的貼標匯入流程（使用 CSV）
- 為什麼我最後選擇讓 LLM 擔任「語意 QA」，而非直接做貼標主體

這一篇，我要回到工程視角，完整說明：**這套系統是怎麼被一層一層組起來的？又為了解決什麼語意上的痛點？**

## 🏷️ 核心目標：讓貼標成為一件可重複、可解釋、可治理的事

這個系統不是做資料探勘的學術研究，也不是為了訓練一個完美模型；它的任務只有一個：

> 給定一篇文章，幫我產生一組資訊密度高、主題辨識度強、可被 Notion 多欄分類使用的標籤集。

要達成這件事，我需要處理掉以下幾種「分類殺手」：

- **語助詞**：例如「出發前」、「整體來說」，無資訊但頻繁出現
- **抽象詞**：如「經驗」、「感覺」、「過程」，模糊難以分群
- **詞彙近義但型式分歧**：如「閱讀反思」與「閱讀心得」，語意接近但難自動合併
- **模型語言偏差**：有些模型偏好短詞、有些則輸出大量片語詞組

這些都是我在系統規劃一開始，就列入的設計考量。

## 🔧 模組組成：語意貼標治理系統的核心組件

這套系統由以下幾個模組組成，每一層都對應一個語意處理問題。

| 模組名稱 | 功能說明 |
|---|---|
| `extract_tags_fusion.py` | 呼叫多模型（KeyBERT、CKIP-BERT 等）並整合標籤、分數與候選集 |
| `tag_cleaner.py` | 清洗停用詞、語助詞、後綴詞等語意雜訊詞 |
| `llm_reviewer.py` | 結構化詢問 LLM 是否應剃除標籤（抽象 / 過渡語 / 主題無關） |
| `review2filter.py` | 根據 LLM 輸出產生最終貼標結果（含移除標籤與原因記錄） |
| `write_back_tags.py` | 同步將 tags 寫入 `.md` metadata 與 `articles.csv` 供匯入 Notion |

每個模組都支援 CLI、具備可單獨執行的範例，也可以串成一條完整的 pipeline。我保留了中間的輸出檔案（如每階段的 CSV），方便逐步檢查與審核，不怕黑箱。

📎 補充：每個模組都可以單獨修改詞庫與參數，如：

- `custom_stop_words.txt`：補上自己不希望保留的詞
- `suffixes.txt`：定義需剃除的尾字（如「區」、「族群」、「過程」）

## 🧹 詞彙治理：讓貼標系統懂得什麼叫「沒意義的詞」

比起模型產生標籤，更難的是判斷「什麼詞不能留」。這就是所謂的語意雜訊處理。

這層詞彙治理包含以下幾種策略：

### 1️⃣ 停用詞清單（Stopwords）

我一開始採用的是 GitHub 上的 stopwords 簡體清單，並透過 OpenCC 轉換為繁體版，再配合 CKIP 的詞性標註，手動擴增了一批本地語境常見但無分類價值的詞彙，例如：

- 某些、這些、事情、過程、方式、狀態、類型、人們、大家⋯⋯

我並額外新增一份 `custom_stop_words.txt`，專門用來補充語意觀察後發現的新噪音詞，並可逐步精煉。

這讓我可以隨著審查經驗演進，滾動式調整「不能當分類」的詞語定義。

### 2️⃣ 後綴詞清除（Suffix Removal）

我發現很多貼標模型會吐出一些「語根 + 結構後綴」的詞，例如：

- 旅程的一部分 → 部分
- 某種想法 → 想法
- 自己的觀察 → 觀察、感受、經驗

這些詞有的可留、有的過度抽象。所以我建立了一份 `suffixes.txt`，專門移除這類後綴，用以：

- 清除那些模糊但語言看似合理的標籤
- 提高語意密度（從「部分」還原為「心理」）

條件是：必須搭配字數長度限制（避免將「地方」從「高雄地方」中誤砍）

### 3️⃣ 語意相似度過濾（semantic similarity filtering）

有些標籤乍看有語意，但其實與主題毫無關聯：

例如：

- 文章在講「大峽谷旅行攝影心得」，模型卻產生了「心情」、「規劃」、「路程」

這些詞語在文字中確實出現過，但若語意距離與標題落差過大，分類上就是雜訊。

我加入了語意嵌入模型（如 zh-embed），計算每個標籤與標題之間的 cosine similarity，若低於門檻（例如 0.2）就視為「主題無關詞」。

這讓整體標籤的主題聚焦度大幅提升，也避免留下無意義的泛詞。

### 4️⃣ POS 結構限定（詞性濾除）

搭配 CKIP 的斷詞與詞性標註，我可以選擇保留：

- 名詞（地點、人名、概念）
- 動名詞（活動）
- 關鍵副詞（若具分類意涵）

並捨棄：

- 助詞、介系詞、語氣詞（例如「著」、「的」、「和」）
- 非實質詞性，如語助構詞

這進一步濾除那些只有語句功能、沒有語意內容的標籤詞。

## 🔁 詞彙治理的本質：讓貼標模組知道「語意密度」是什麼

這些策略背後都有一個核心邏輯：

> 好的標籤不是詞彙看起來專業，而是它能幫你把一群主題相近的文章拉在一起。

這也就是為什麼我重視語助詞剃除，因為「這次旅行」、「部分想法」看似人話，但分類貢獻其實是 0。

你想要的標籤，是能讓 Notion 裡的內容真正可以群聚與篩選，而非當裝飾。

## 🤖 LLM 審查層：讓 AI 成為一個能被審計的語意審查員

如前面提到，LLM 並不適合主導貼標這件事，因為它的語言直覺雖強，但邏輯判準缺乏一致性。因此，我重新定位它的角色，讓它只負責判斷以下幾件事：

- 某個標籤是否**過於抽象**
- 某個詞是否只是**語意過渡或語助詞**
- 該詞語是否與文章主題**毫無語意交集**

只要能回答這三件事，LLM 就能補足語言模型難以形式化處理的那一層「語感判斷」。

### 🧩 回應設計策略：結構化格式讓 LLM 輸出更可控

我使用的 prompt，不是單純問它「這個標籤要不要留」，而是要求回應一段可結構化、可機器解析的 JSON 格式：

```json
{
  "should_filter": true,
  "reasons": ["抽象", "語助詞"],
  "explain": "該詞語為段落過渡常用詞，未提供明確分類資訊"
}
```

這樣一來，我可以：

- ✅ 在後處理時機器自動判斷是否需要移除該標籤
- ✅ 統計標籤被移除的主要類型（可供可視化分析）
- ✅ 將 `explain` 回應寫入審查報表中，提高系統可審計性

這是我在 `llm_reviewer.py` 中完成的，並搭配 `review2filter.py` 將應剃除標籤進行移除處理。

### 🧠 Prompt 設計要點

我精心設計了 prompt 模板，以 few-shot 方式提供模型學習語意密度低的語詞範例：

```text
你是語意標籤審查員，請根據下列輸入判斷此詞是否適合作為分類標籤。

考量條件：
- 詞語是否具體而非抽象（如「美國國家公園」vs「規劃」）
- 是否僅為語意過渡詞（如「出發前」、「整體來說」）
- 是否與主題無明顯語意連結

請輸出 JSON 格式，包含 should_filter、reasons、explain 三項

範例：
tag: 出發前 / title: 我的大峽谷攝影紀錄
→ { should_filter: true, reasons: ["語助詞", "抽象"], ... }
```

這樣的設計能讓 LLM 更專注於「語意品質」的判斷，而不是字面意義的好壞。

### 🔍 實際成效

我在審查報表中觀察到，被剃除最多的理由集中於三大類：

- 抽象詞（如：感覺、方向、過程）
- 語助過渡詞（如：出發前、這次旅程）
- 主題無關詞（如：完成、整理、部分）

這代表 LLM 的確在補上機器模型無法處理的語意縫隙，讓最終輸出的標籤更具語意密度與分類價值。

## 🔎 每層功能摘要

| 模組名稱 | 角色 | 核心功能 |
|---|---|---|
| `extract_tags_fusion.py` | 初步貼標 | 結合 KeyBERT、CKIP-BERT、zh-embed 等模型產生貼標建議 |
| `tag_cleaner.py` | 詞彙治理核心 | 清除語助詞、後綴詞、停用詞、控制詞長與符號，建立語意密度範圍 |
| `llm_reviewer.py` | 智能審查員 | 使用 LLM 判斷詞語是否抽象、無主題語意、語助詞，並結構化回應 |
| `review2filter.py` | 可追蹤剃除器 | 根據審查結果移除不合格標籤、統整留存清單與移除記錄 |
| `write_back_tags.py` | 同步轉寫器 | 將最終標籤寫回 markdown metadata 與 `articles.csv` 中，便於 Notion 匯入 |

## 🎯 結語：治理貼標，其實就是在治理內容的可用性

我一開始設計這套貼標治理系統，是為了剃除那些無意義的標籤；但一路走到最後，我才發現，它其實不只是處理詞語的問題。

它處理的是：**我們怎麼看待自己寫下來的知識與內容**。

當我能清楚知道每一篇文章談的是什麼主題，它與哪些詞彙相關、可以如何被分群、能不能被找出——這不只是分類能力的提升，而是知識再利用能力的提升。

最終，我不再是單純在做「語言模型應用」，而是建立了一個個人語意治理的框架：

- 每一次清洗標籤，其實是在問：「這個詞真的有資訊價值嗎？」
- 每一次審查詞彙，其實是在定義：「這段話到底要讓我記住什麼？」

我認為這才是語意貼標治理系統真正的價值所在。

而所有這些工作，不只幫我整理過去的資料，也讓我未來在寫作時，更知道自己想留下什麼。

貼標，不是結束；是知識系統真正可組織、可擴展的開始。
